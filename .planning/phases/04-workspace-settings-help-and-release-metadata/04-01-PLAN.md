# Plan 01: Build The Settings Hub And Local Workspace Preferences

<objective>
Turn `/settings` into a four-section workspace hub that keeps editable scope narrow, uses browser-local persistence for workspace preferences, and clarifies that source labels support demo workflow context rather than real integrations.
</objective>

<must_haves>
<truths>

- D-01: one settings page shell with section switching remains the default shape.
- D-09: workspace preferences stay in `localStorage`, not the shared database.
- D-12: Phase 4 does not add schema or backend settings infrastructure.
- D-22: assistant controls in this phase stay small and must not imply browser control over secrets or deployment config.
  </truths>
  </must_haves>

<threat_model>

- T-04-01: New workspace settings could look like real multi-user admin data and mislead reviewers. Mitigation: label them as demo or browser-local preferences and keep scope narrow.
- T-04-02: Reset behavior could accidentally touch shared seeded data. Mitigation: reset only the namespaced `localStorage` key and reload the view state from defaults.
- T-04-03: Source-management copy could imply LinkedIn, WhatsApp, or phone integrations exist. Mitigation: add explicit explanatory copy around source labels and intake context.
  </threat_model>
